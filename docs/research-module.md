# 调研模块设计

## 1. 模块判断

调研模块应该单独存在，但它的沉淀结果进入知识库。

原因：

- 知识库是仓库：已经确认、整理、可复用的资料。
- 调研是工作流：围绕一个问题收集资料、验证假设、比较方案、形成结论和行动。

一句话定义：

> 调研模块是把不确定问题变成可执行判断的 AI 工作台。

也可以理解为：

> 调研模块是知识生产线，知识库是知识仓库。

## 2. 调研工作流

一个 Research 应从问题开始，而不是从资料开始。

建议流程：

1. 提出研究问题。
2. 写下当前假设。
3. 收集资料和来源。
4. AI 阅读、摘要和提取发现。
5. 比较观点、证据和反例。
6. 形成综合结论。
7. 转化为任务、决策、知识或 Playbook。

## 3. Research 对象

每个调研任务建议包含：

- 标题
- 研究问题
- 所属项目
- 调研类型：产品、市场、竞品、技术、用户、商业、内容
- 当前假设
- 资料池
- 关键发现
- 反例与风险
- 综合结论
- 后续行动
- 需要沉淀的知识
- 关联 Playbook
- 状态：草稿、收集中、分析中、已结论、已沉淀

## 4. 资料池

资料池支持：

- 链接
- 文件
- 笔记
- 截图
- 对话
- 搜索结果
- 会议记录

每条资料建议记录：

- 来源
- 摘要
- 可信度
- 关键摘录
- 关联假设
- 关联项目
- 是否已沉淀到知识库

## 5. 调研产出

调研结束后不应该只留下长文档，而是要能分发到系统各处。

可转化为：

- 知识卡片：进入知识库。
- 决策记录：进入项目。
- 后续任务：进入待办。
- 新项目：进入项目中心。
- Playbook：进入可复用流程。
- 自动化需求：进入自动化中心。

## 6. AI-heavy 版本

调研模块适合拆成多个 Agent。

- 搜索 Agent：找资料。
- 阅读 Agent：摘要和提取重点。
- 对比 Agent：做竞品矩阵或方案对比。
- 反驳 Agent：找风险、反例和盲点。
- 综合 Agent：写 research memo。
- 执行 Agent：把结论变成任务、知识和项目更新。

关键不是让 AI 输出一篇漂亮报告，而是让 AI 帮用户完成从不确定到可执行判断的闭环。

## 7. v1 已实现（2026-07）

围绕「保存链接、截图、摘要、总结」的最小闭环落地（T-6）：

- **数据**：`research`（R-x：title + question 研究问题 + conclusion 结论 + 状态 收集中/已有结论/已归档 + 项目/任务关联）+ `research_items` 资料卡（title + url + image 截图 + body 摘要/爽点 + tags + rating 0-5 星）。附件存 `data/files/`。
- **CLI**：`opc research`（别名 `rs`）`add / list / show / item / edit-item / rm-item / conclude / edit / distill / rm`；截图支持 `--image` 本地导入和 `--image-url` 网络抓取；`distill` 把结论（带精选参考）转成知识库条目并归档——对应「产出分发」里的知识卡片路径。
- **Web**：`#/research` 列表卡片（封面图 + 状态 + 资料数）→ 详情（研究问题 / 资料池网格 / 结论区），资料卡点开看大图和全文，支持上传/粘贴截图、星级打分、一键沉淀。
- **未做**（等真实使用撞出需求再加）：假设（hypothesis）字段、可信度、反例与风险区、Playbook/决策记录转化、多 Agent 编排。当前用 question/body 的 Markdown 自由承载。
