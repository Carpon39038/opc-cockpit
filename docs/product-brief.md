# OPC Cockpit 产品讨论整理

## 1. 产品判断

OPC Cockpit 应该是一个行动驾驶舱，而不是信息展示面板。

它每天要回答的问题是：

- 我现在最该做什么？
- 为什么是这件事？
- 做这件事需要哪些上下文？
- 哪些可以交给 AI 或自动化？
- 做完以后沉淀到哪里？

## 2. 三层结构

### 人的执行层

这是用户每天真正使用的部分。

- 今日作战室：今日重点、时间块、关键任务、等待事项。
- 待办与日历：任务、提醒、周期任务、日程。
- Inbox：随手捕捉想法、链接、语音、会议纪要、聊天片段。
- 项目状态：每个项目当前目标、进度、阻塞、下一步。
- 注意力管理：区分深度工作、沟通、整理、交付。

### 公司的记忆层

这是让一人公司不靠脑子硬扛的部分。

- 知识库：文档、笔记、网页、研究资料、会议记录。
- 决策日志：记录当时为什么这么选。
- Playbook/SOP：重复工作流程。
- 联系人/CRM：客户、合作方、潜在线索、跟进记录。
- 资产库：文案、产品素材、合同、报价、案例、品牌资料。

### AI 增强层

这是超级人体的核心。

- Skill/Playbook 管理：把常用 AI 工作流沉淀下来。
- Agent 任务队列：让 AI 帮用户跑研究、整理、生成、复盘。
- 上下文包：每个任务自动拉取相关项目、知识、历史决策、联系人。
- 自动化：日报、周报、会议总结、待办生成、知识更新。
- 审核台：AI 做了什么、依据是什么、需要用户确认什么。

## 3. 建议模块

### 驾驶舱首页

首页应该是执行入口，而不是大屏展示。

建议包含：

- 今日重点
- 日历时间轴
- 待办列表
- Inbox
- 正在推进的项目
- 等待他人的事项
- AI 建议的下一步
- 最近调研和知识更新

### 项目中心

项目是整个系统的主轴。任务、知识、调研、决策、联系人和 Playbook 都应能挂到项目上。

每个项目建议包含：

- 目标
- 当前状态
- 下一步
- 阻塞与风险
- 关联任务
- 关联日程
- 关联知识
- 关联调研
- 决策记录
- 可用 Playbook

### 任务与日历

任务管理不要只做 todo，而要和上下文绑定。

每个任务应支持：

- 所属项目
- 截止时间
- 预估耗时
- 优先级
- 相关知识
- 相关联系人
- 可调用 Playbook
- AI 生成的上下文包

### Inbox

Inbox 是低摩擦输入口。

所有零散输入先进入 Inbox，再分流为：

- 任务
- 知识
- 项目
- 调研
- 联系人
- Playbook
- 决策
- 归档

### 知识库

知识库是工作记忆，不只是文档仓库。

关键能力：

- 来源记录
- 自动摘要
- 标签与项目关联
- 语义搜索
- 引用与可信度
- 过期提醒
- 从调研结果沉淀知识卡片

### Playbook / Skill

第一版建议先叫 Playbook，不要过早做成复杂 Skill 市场。

一个 Playbook 可以包含：

- 用途
- 输入
- 步骤
- 输出格式
- 适用项目
- 所需知识
- 所需工具或权限
- 示例
- 运行记录
- 质量检查标准

常见 Playbook：

- 写周报
- 整理会议
- 生成 PRD
- 做竞品研究
- 客户跟进
- 内容选题
- 项目复盘

## 4. 产品原则

- 以项目为中心，不以文件夹为中心。
- 以下一步行动为中心，不以信息展示为中心。
- 所有知识都要能被任务调用。
- 所有调研都要能转化为判断、任务或知识。
- 所有自动化都要有日志、可解释、可确认。
- Skill 管理先轻量，等重复流程稳定后再增强。
